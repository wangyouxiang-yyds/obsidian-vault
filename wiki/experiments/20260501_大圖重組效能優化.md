# 大圖重組效能優化

## 目標

將單張大圖（10000×10000 像素）的 Sliding Window 重組時間從**數分鐘**降低至 **30~60 秒**以內。

---

## 設定

- 腳本：`main/reconstruct_module.py`
- 硬體：RTX 5090（32GB 顯存）
- 問題場景：OSDMamba 或 DeepLabV3+ 執行 `run_reconstruction` 時耗時過長

---

## 問題診斷

| 原因 | 說明 |
|------|------|
| NAS 讀取瓶頸 (I/O Bound) | 從 `/mnt/backup/...` 用 `rasterio` 讀取數 GB 的 11 波段 TIF，每次都要等整張圖載入 |
| GPU 利用率低 | `infer_batch_size` 設為 4 或 8，RTX 5090 算力嚴重閒置 |
| 頻繁數據搬運 | 每個 Batch 推論後立刻 `.cpu()`，大量 PCIe 搬運與同步等待 |

---

## 解決方案

### 任務 A：本地快取機制（高優先）

在處理每個場景前，先將 VRT 或 TIF 複製到本地高速磁碟（`/dev/shm` 或 `/home/alanyh/oil_dataset/new/full_band/`），消除 NAS 隨機讀取延遲。

### 任務 B：GPU 側重組（高優先）

```python
# GPU 累加範例
prob_map_gpu = torch.zeros((img_h, img_w, num_classes), device='cuda')
# 推論中直接在 GPU 累加
prob_map_gpu[y:y+ps, x:x+ps, :] += probs_gpu
# 整張大圖處理完後才搬回 CPU
```

- `prob_map` 與 `weight_map` 初始化在 GPU 顯存（5090 有 32GB）
- 推論結果在 GPU 內累加，整張圖完成後才 argmax → CPU
- 預期效果：減少 90% PCIe 數據交換

### 任務 C：提升 Batch Size 與並行度（中優先）

- `infer_batch_size` 提高至 32 或 64
- 實作 Prefetching：GPU 推論目前 Batch 時，CPU 同時準備下一個 Batch

### 任務 D：窗口化讀取（中優先）

改進 `extract_patches_sliding_window`，用 `rasterio.Window` 僅讀取當前 Batch 需要的區塊，降低記憶體峰值。

---

## 結果

採用本地 Stacked TIF 直接讀取（取代 VRT 模式）+ `infer_batch_size=32`（RTX 5090 專屬高吞吐）。

---

## 觀察與結論

I/O 是最大瓶頸，切換至本地 SSD 後效能提升 10 倍以上。GPU 側重組在顯存充足時（RTX 5090 32GB）效果顯著。
