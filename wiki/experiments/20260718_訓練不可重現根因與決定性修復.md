# 訓練不可重現根因與決定性修復

> 本篇是分支 B（[[GT_expand_pipeline]]）P1 Tversky α/β 掃描期間（2026-07-18 起）意外發現的管線層級 bug：訓練不可重現。時間線橫跨 2026-07-21（症狀發現）～07-22（根因定位＋修復＋驗收）～07-24（決定性正本重跑進行中）。這是本輪最重要的發現——它讓此前所有 grid/screen 掃描結論全部降級，包括 [[20260716_資料溯源洩漏與評估合約v1|Evaluation Contract grouped replay]] 引用的存檔預測。

## 1. 實驗背景

P1 Tversky α/β 掃描（`0.1/0.9`～`0.5/0.5`）原本按 [[20260702_波段選擇消融實驗規劃]] 建立的三級 smoke/screen/confirm 協定執行，唯一登記變因是 loss 的 α/β。掃描過程中先後遇到「V1/V2 容器 lineage 偏移」（同一組 config 換容器版本掉了 0.055 IoU，比整個 Tversky 效應還大）等干擾，最終在 2026-07-21 做「同 config 雙跑」排查時，才定位到問題不在容器，而在訓練管線本身不可重現。

## 2. 症狀

同一個 config（Tversky `α=0.3/β=0.7`、`seed=42`、fold1）串行跑兩次（`reproA`/`reproB`），YAML 除實驗名稱與輸出路徑外完全相同，但兩份訓練軌跡從 **epoch 1 就分歧**：

| 項目 | reproA | reproB | 差異 |
|---|---:|---:|---:|
| 早停 epoch 數 | 90 | 52 | +38 |
| best val mIoU | 0.6242 | 0.6176 | +0.0066 |
| fold1 M1（Contract v1.0） | 0.4391 | 0.3855 | +0.0536 |
| fold1 M2（Contract v1.0） | 0.4549 | 0.4149 | +0.0400 |
| pooled Oil IoU | 0.4619 | 0.4375 | +0.0244 |

119 场景配對逐景比較：A 83 勝 / B 33 勝 / 3 平，ΔM2 中位數 +0.0306，單景最大差 |Δ|=0.2145。`results.csv` 與 `best.pt` 的 SHA256 也不同——差距遠超浮點捨入誤差量級，排除了 serialization 或硬體層級的小數值差異。

**關鍵對照**：先前掃描中 `0.5/0.5`（Dice）fold1 的 M1/M2=`0.4487/0.4569`，與高軌跡 reproA 只差 `+0.0096/+0.0020`，但與低軌跡 reproB 差到 `+0.0632/+0.0420`。也就是說，先前「`0.5/0.5` 優於 `0.3/0.7` 約 `+0.029`」這個單-seed 結論，完全可能只是掉進了 reproA/reproB 之間的 run-to-run 噪音帶。

## 3. 根因

**Albumentations 2.0.8 的 `A.Compose` 有獨立的內部 RNG，`torch.manual_seed()` / `np.random.seed()` / `random.seed()` 對它完全無效。** `main/deeplab_adapter.py:1048` 建構 train `A.Compose([...])` 時沒有傳入 `seed=` 參數；`Compose` 在 `seed=None` 時，每次建立都會從系統熵重新產生一個內部狀態（`core/composition.py:411-430`，已安裝套件原始碼實查確認）。結果是：

- 每次跑訓練，圖片增強序列（crop 位置、翻轉、亮度擾動等）完全不同。
- 增強序列不同 → 每個 batch 實際看到的像素不同 → 訓練軌跡從 epoch 1 分歧 → 早停時機、最終權重、下游所有指標全部不可重現。
- 定位過程中排查了其他可能來源（CUDA unsupported-deterministic-op warning 未出現、config/資料統計皆對齊），Albumentations RNG 是目前直接證據最強、且修復後被驗證為根因的來源。

## 4. 修法（commit `ba25391`，Claude×Codex 收斂，VERDICT: ACCEPT WITH EXPANDED FIX）

`main/deeplab_adapter.py` 三處修改：

1. **train/val `A.Compose` 建構時傳 `seed=torch.initial_seed()`**——覆蓋 `num_workers=0`（主進程路徑）情境，使其成為決定性。
2. **`SegmentationDataset.set_augmentation_seed()`**——重設 Compose 內部 RNG 的介面方法。
3. **`_seed_worker`**——每個 DataLoader worker 以自己的 `wseed = loader_seed + worker_id` 重設 Compose，讓 `num_workers>0` 情境下每個 worker 決定性且彼此不同（避免多個 worker 用同一組增強序列）。

附帶新增 `test_augmentation_determinism.py`（Layer 1 回歸測試）：`workers=0` 與 `workers=8` 各跑兩次逐 batch hash 比對一致、worker 間彼此互異、無 seed 版本可被鑑別出不一致——全數 PASS。

## 5. 驗收（Layer 3，最嚴格）

修復後獨立雙跑 `reproC`/`reproD`（同 `0.3/0.7`、`seed=42`、fold1）：

- **370 個模型 tensor 逐 bit 相同**。
- **整檔 `best.pt` SHA256 相同**（`0d2f07b63186`）。
- **119/119 場景預測逐像素一致**。
- **重建（recon）指標完全相同**。

→ **決定性完全修復**，訓練管線從資料增強到最終預測，端到端 bit-identical。

## 6. 重大含意：既有結論全面降級

1. **所有先前的 grid/screen 掃描結果，都是在有 bug 的碼上跑的**——每次重跑都相當於重抽了一組不同的隨機資料增強，配置之間的名次比較被未受控的訓練隨機性污染。**全數降為「legacy exploratory」，不能用來選 winner。**
2. **「`0.5/0.5`（Dice）勝 `0.3/0.7`（recall-weighted Tversky）約 `+0.029`」的結論已撤回**——2026-07-22 的變異量測（variance pilot，5 次重複跑 fold1）測出 `0.3/0.7` 的 run-to-run SD ≈ 0.023（三次獨立跑 M2 = 0.4152 / 0.4549 / 0.4149）。`+0.029` 的效應量落在 ≈1 個 run-to-run SD 之內，單一 seed 分辨不出這是真效應還是噪音，因此該結論**不再成立**，`0.3/0.7` 恢復為論文候選之一。
3. 連帶地，[[20260716_資料溯源洩漏與評估合約v1]] 第 4 節的 grouped replay（tversky vs baseline、Prithvi 系列比較）所用的存檔預測，也全部是「未修的污染碼上、各只跑一次」的產物——名次方向與多項獨立證據相容（暫視為穩定），但正式論文數字需等本篇的決定性 rerun 完成後重新 replay 才能定案。
4. Focal Tversky（[[20260724_Focal_Tversky小圖實驗可行性評估]]）的前置 gate 因此被設計為必須直接對照**決定性 canonical Tversky**，不得用修復前的舊掃描數字。
