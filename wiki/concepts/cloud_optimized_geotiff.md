# Cloud Optimized GeoTIFF (COG)

## 一句話定義

COG 是一種 GeoTIFF 儲存格式，把像素資料切成 256×256 的小方塊（tile）存放，讓硬碟可以**直接跳到指定區塊讀取**，不需要繞路讀整行再切。

---

## Strip TIF vs COG TIF：根本差別

兩者的差別不在檔案內容，而在**資料在磁碟上的擺放方式**，以及隨之而來的 index 粒度。

### Strip TIF — index 只有「行」的粒度

```
磁碟排列：
┌─────────────────────────────────────────┐
│ row 0   : [px][px][px][px]...[px][px]   │ ← 10980 個像素連續排
│ row 1   : [px][px][px][px]...[px][px]   │
│ row 2   : [px][px][px][px]...[px][px]   │
│ ...                                     │
│ row 512 : [px][px][px][■■■][px]...[px]  │ ← 只能跳到行首
│ row 513 : [px][px][px][■■■][px]...[px]  │   col 1024 在哪？不知道
│ ...                                     │   必須把整行讀進來再切
│ row 767 : [px][px][px][■■■][px]...[px]  │
│ ...                                     │
└─────────────────────────────────────────┘
讀了 256 × 10980 = 2,810,880 個像素
你要的只有 256 × 256 = 65,536 個像素  → 浪費 43×
```

Strip TIF 的 index 只記錄「第幾行從磁碟哪個 byte 開始」。你要讀 col 1024~1279，硬碟跳得到 row 512 的行首，但不知道那一行裡 col 1024 在第幾個 byte，只能把整行讀進記憶體再切。

### COG TIF — index 有「tile」的粒度，直接跳過去

```
磁碟排列：
┌──────┬──────┬──────┬──────┬──────┬──────┐
│t(0,0)│t(0,1)│t(0,2)│t(0,3)│t(0,4)│ ...  │
├──────┼──────┼──────┼──────┼──────┼──────┤
│t(1,0)│t(1,1)│t(1,2)│t(1,3)│t(1,4)│ ...  │
├──────┼──────┼──────┼──────┼──────┼──────┤
│t(2,0)│t(2,1)│t(2,2)│t(2,3)│t(2,4)│ ...  │ ← row 512~767
├──────┼──────┼──────┼──────┼──────┼──────┤    col 1024~1279
│ ...                                      │    = tile(2,4)
└──────────────────────────────────────────┘

index 直接記錄 tile(2,4) 在磁碟的第幾個 byte
→ 硬碟直接跳過去，讀 256×256，完成

讀了 256 × 256 = 65,536 個像素  → 剛好你要的量
```

COG 的 index 在檔案開頭（Ghost Header）記錄每個 tile 的起點 byte，讀任何區塊前只需一次 seek，直接取目標資料。

---

## 為什麼 Strip TIF 是預設格式

Strip TIF 是 GeoTIFF 設計年代的產物，當時衛星影像的主要用途是**整張圖全部讀取**（顯示、全圖處理），循序讀取整行效率很好。COG 是後來專門為**隨機讀取小區域**設計的，剛好對應深度學習 patch 讀取的使用場景。

---

## COG 的其他特性

**Overview（金字塔縮圖）**

COG 預先儲存多層縮圖供快速瀏覽：

```
Level 0（原始）：10980 × 10980
Level 1（1/2）：  5490 × 5490
Level 2（1/4）：  2745 × 2745
Level 3（1/8）：  1373 × 1373
```

QGIS 縮放瀏覽時直接讀低層縮圖，不需讀原始資料再縮小，開圖速度快很多。訓練時用不到這個。

---

## 在本專案的應用

Sen2Like 輸出預設為 Strip TIF，訓練時每次讀 256×256 patch 觸發 Strip 的整行讀取，造成嚴重 I/O 浪費：

| 指標 | Strip TIF | COG TIF |
|---|---|---|
| 讀 256×256 patch 的實際 I/O 量 | 256 × 10980 像素（43× 浪費） | 256 × 256 像素（精確） |
| 單波段 256×256 讀速（B03） | 397 ms | **43 ms**（9.1×） |
| 8-band VRT 全場景讀速 | 1600 ms | **450 ms**（3.6×） |
| 訓練 epoch 速度 | 8–9 min | **~3 min** |

**轉換腳本**：`/home/alanyh/oil_dataset/new/full_band/convert_to_cog.py`
- In-place 轉換，先寫 `.cog_tmp.tif` 驗證後再覆蓋原檔
- `is_already_tiled()` 自動跳過已轉換的檔案，可全目錄重跑
- B01（60m）跳過，其餘 7 個 band 都轉

---

## 與 VRT 的關係

VRT 和 COG 是**不同層次**的東西：

```
訓練程式
    ↓ rasterio.Window 讀 256×256
  VRT（XML 描述檔，只是指標）
    ↓ 指向
  底層 TIF 檔案  ← Strip / COG 的差別在這裡
```

VRT 自己沒有像素資料，快不快取決於它指向的 TIF 格式。COG 轉換後 VRT 仍然存在，只是底層變快了。
